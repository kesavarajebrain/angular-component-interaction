Signal

Signals are a new, built-in way for Angular to track state changes and automatically update your UI — without relying on Zone.js or the old change detection system.

🧠 Simplified Explanation

Before Signals:
  • Angular had to walk the whole component tree (change detection) to see what changed.
  • It relied on Zone.js to know when to start checking.
  • This caused unnecessary checks — even when nothing really changed.
    • With Signals:
  • Each piece of data tells Angular directly when it changes.
  • Angular doesn’t need to check everything — only what’s affected.
  • No Zone.js required. No over-checking.

So now, Angular becomes fine-grained reactive — like Reactivity in Solid.js or Vue 3.

Analogy — Think of It Like an Electric Wire
  • Imagine every variable (like count) is connected to your UI via a signal wire ⚡
  • When you update that signal, it sends an electrical pulse 💥 to Angular.
  • Angular immediately knows:
    “Hey, count changed — update only that part of the UI.”
  • ✅ No need to run a full change detection cycle.
  • ✅ No need for Zone.js.
  • ✅ Lightning fast updates.
Count : {{count()}}

in signal we use function as interpolation because now count is not a normal variable anymore — it’s a Signal object, and to get its current value, you call it like a function — count().

                    {{tsCode}}
when we write like this - count = signal(0);
This creates a reactive wrapper object — not just a number.
That signal(0) returns an object that looks like this (internally):

like below:
  value: 0,
  set(newVal) curly braces ... ,
  update(fn) curly braces ... ,
  mutate(fn) curly braces ... ,
  // and a getter function (callable like count())

So when you call count(), you’re reading the signal’s value.

✅ count() → means “give me the current value inside the signal”
✅ count.set(5) → means “replace it with 5”
✅ count.update(v => v + 1) → means “increment the current value”

| --------------- | ------------------ | -------------------------

| Reading value   | `{{ count }}`      | `{{ count() }}` 
| Updating value  | `this.count++`     | `this.count.update(v => v + 1)`

Signal is a replacement of the older change detection strategy?

⚙️ 1️⃣ Before Signals: The “Old World” 🌍

Components + Zone.js + Change Detection

  • Angular uses Zone.js to know when async events happen (click, HTTP, etc.).
  • Then it runs Change Detection over the entire component tree.
  • Default mode → check everything
  • OnPush mode → check only components with changed @Input() or manually marked
  • To force updates, we used detectChanges() or markForCheck()
✅ Works fine
❌ But checks too much → can slow down large apps
❌ Relies heavily on Zone.js (a global monkey patch)
❌ Hard to optimize — too much auto-magic

⚡ 2️⃣ With Signals: The “New World” 🚀

Instead of Angular checking everything,
Signals tell Angular directly when and what changed.

  • No Zone.js.
  • No global change detection
  • No need for detectChanges() in most cases
  • No need for Default/OnPush configuration
  • Everything happens automatically & precisely 🎯

Signals are the modern replacement for the whole reactive + CD + Zone system

But... Angular still supports the old system (for now)
