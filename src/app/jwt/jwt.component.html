JSON Web Token (JWT)

A compact, URL-safe string that represents claims about a user, signed by the server so it can’t be tampered with. (மாற்ற முடியாதது)

JWT is not securing the frontend
JWT is: A backend mechanism to safely identify the caller and decide what they are allowed to access. Frontend just helps in transporting it.

JWT Structure

A JWT always looks like:
xxxxx.yyyyy.zzzzz

Three Base64-encoded parts separated by dots:
HEADER.PAYLOAD.SIGNATURE

▶ Header
Metadata about the token: 

{
  "alg": "HS256",
  "typ": "JWT"
}
Means:
alg → signing algorithm
typ → token type

▶ Payload (Claims)
This contains the actual data:

{
  "sub": "12345",
  "email": "kesava@example.com",
  "role": "USER",
  "iat": 1700000000,
  "exp": 1700003600
}

▶ Signature (the protection)
Created like:

HMACSHA256(
  base64(header) + "." + base64(payload),
  SECRET_KEY
)

Only the server knows SECRET_KEY
So if someone edits the payload: "role": "ADMIN"
→ signature becomes invalid
→ server rejects the token 💥

How JWT Works on the Server?

Step 1 – User logs in

POST /login
{ email, password  }

Step 2 – Server verifies credentials

If OK:
* creates JWT
* signs it
* sends it back

{
  accessToken: "eyJhbGciOi..."
}

Step 3 – Browser stores token

Usually:
* HttpOnly cookie ✅ (best)
* OR memory
* OR localStorage (less secure)

Step 4 – Browser sends token on every API call

 Authorization: Bearer eyJhbGciOi...

Step 5 – Server middleware validates:

* signature
* expiry
* issuer
* audience

If valid → continue
If not → 401 Unauthorized

Step 6 – Refresh token:

Now You have:
1️⃣ Login → get access token (2 min)
2️⃣ Store in localStorage
3️⃣ Interceptor sends it for protected APIs

All good for learning ✅
Now the problem:
⏱ After 2 minutes → token expires → user should NOT be forced to login again,
in real world application once user loggged in , untill they manually logout the application, still we have to keep their session in particular timing based on their activity since that is the secure application means.
That’s where refresh token comes in.

When should refresh token API be called? 

| Strategy                   | Used in real apps?   | Description                        |
| -------------------------- | -------------------  | ---------------------------------- |
| Call before expiry (timer) | ❌ Rare             | Check expiry & refresh proactively |
| Call after 401             | ✅ MOST COMMON      | Refresh only when API fails        |

👉 Enterprise standard = refresh on 401
Why?

  • no unnecessary API calls
  • simpler logic
  • scalable (இது செயல்திறன் குறையாமல் பெரிய அளவிலான தேவைகளைக் கையாளுவதை உறுதி செய்கிறது)
⭐ Real Enterprise Flow

User is working normally:
API call → access token valid → ✅

After 2 minutes:
API call → 401 Unauthorized

Interceptor catches it:
→ calls /refresh
→ gets new access token
→ retries original API automatically

User doesn’t notice anything 😄
                
🧠 Why This Is Better?

Imagine:
10,000 users

token refresh every 1 minute
💥 huge unnecessary load

Instead:
Refresh only when API is actually used.
                
LOGIN API
   ↓
User sends credentials (email/password)
   ↓
Server validates the user
   ↓
Server generates two tokens
      • accessToken  (short lifetime – 2 min in your case)
      • refreshToken (long lifetime – hours/days)
   ↓
Server returns accessToken + refreshToken
   ↓
Frontend (Angular) stores them
      accessToken  → used for API authentication
      refreshToken → used only for refreshing the session
   ↓
User calls protected APIs
   ↓
Interceptor attaches accessToken automatically
Authorization: Bearer accessToken
   ↓
Server verifies JWT
   ↓
If token valid → API works normally
   ↓
After 2 minutes → accessToken expires
   ↓
Next protected API call sends expired token
   ↓
Server detects expired token
   ↓
Server returns 401 Unauthorized
   ↓
Angular interceptor catches the 401 error
   ↓
Interceptor calls /refresh API (IF refresh API fails → logout user)
   ↓
Payload contains stored refreshToken
   ↓
Server verifies refreshToken
      • check if token exists
      • check if token is valid
      • check if token belongs to user
   ↓
If valid → server generates NEW accessToken
   ↓
Server returns new accessToken
   ↓
Interceptor stores the new accessToken
   ↓
Interceptor retries the original failed request
   ↓
Retry request now includes the new accessToken
   ↓
Server verifies new token
   ↓
Protected API succeeds
   ↓
User never notices this entire process
Authentication Vs Authorization

Authentication:
👉 உங்களுடைய அடையாளத்தை உறுதி செய்வது
(Verifying who you are)

Authorization:
👉 உங்களுக்கு என்ன செய்ய அனுமதி இருக்கிறது என்பதை தீர்மானிப்பது
(Deciding what you are allowed to access)

Interceptor
