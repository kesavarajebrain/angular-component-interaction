🚫 4. canMatch

🔵 With canMatch

User → /admin
   ↓
Route matching starts
   ↓
canMatch runs
   ↓
false → STOP ❌
   ↓
Module NOT loaded
                

🟢 With canActivate

User → /admin
   ↓
Route matched
   ↓
Module loaded 📦
   ↓
canActivate runs
   ↓
false → block ❌
                
                
canMatch → prevents loading
canActivate → prevents entering

✅ Correct version of your thinking:

canActivate and canMatch are similar in purpose (both guard routes),
but different in timing.

canMatch → gate security
canActivate → door security
            

{
  path: 'admin',
  canMatch: [roleGuard],       // 🚫 block early - say for example this guard check the roles
  loadChildren: () => import('./admin.routes'),
  canActivate: [authGuard]     // 🔐 extra safety
}
        
🧠 Why both?
canMatch → performance + early block
canActivate → security + final check
⚙️ 5. runGuardsAndResolvers

🧠 What is runGuardsAndResolvers?
Controls WHEN guards & resolvers should run again
👉 By default, Angular tries to reuse the same component
👉 So guards/resolvers may NOT run again

⚠️ Problem (Real Scenario)
🎯 Example: User Details Page 

/user/1 → shows User 1
/user/2 → should show User 2

❌ Without runGuardsAndResolvers

👉 Angular behavior:

Same component reused
↓
Guard/Resolver NOT triggered again ❌
↓
Old data still shown 😱

Scenario: Banking App 💰

You have:

/account/1001 → Account A
/account/2002 → Account B

Without this config:
User switches account ID
↓
Component reused
↓
Old account data shown ❌
✅ Solution 

{
  path: 'user/:id',
  loadComponent: () => import('./user.component'),
  runGuardsAndResolvers: 'paramsChange'
}

🧠 What this does
When :id changes → run again

👉 Now:
/user/1 → /user/2
↓
Guard runs again ✅
Resolver runs again ✅
API call happens again ✅
            
With this:

{
  path: 'account/:id',
  resolve: { account: accountResolver },
  runGuardsAndResolvers: 'paramsChange'
}

👉 Now:
Account ID changes
↓
Resolver runs again
↓
New account data fetched ✅
⚙️ Options (Important)
1️⃣ 'paramsChange' (MOST USED)
runGuardsAndResolvers: 'paramsChange'

👉 Runs when:
Route params change (/:id)
            
2️⃣ 'always'
runGuardsAndResolvers: 'always'

👉 Runs every time:
Even same route click
            
3️⃣ 'pathParamsChange'

👉 Only path params (rare usage)
Route.ts 
{
  path: 'user/:id',
  loadComponent: () => import('./user.component'),
  resolve: { user: userResolver },
  runGuardsAndResolvers: 'paramsChange'
}
Resolver 
export const userResolver = (route: ActivatedRouteSnapshot) => {
  const id = route.paramMap.get('id');
  console.log('API CALL for ID:', id);
  return fetchUser(id);
};

🧠 9. matcher
🎯 Scenario: Dynamic URL Format

You want to support:
/profile-123
/profile-456


// matcher.ts

import { UrlSegment } from '@angular/router';

export function profileMatcher(url: UrlSegment[]) {

  if (url.length === 1 && url[0].path.startsWith('profile-')) {

    const id = url[0].path.split('-')[1];

    return {
      consumed: url,
      posParams: {
        id: new UrlSegment(id, {})
      }
    };
  }

  return null;
}
