ngAfterContentChecked()

🔹After every change detection cycle, after it has projected external content into the component.

🔹It runs every time Angular checks the content projected via ⮜ng-content⮞ for updates.

⚙️ When Does It Run?

1️⃣ It runs after ngAfterContentInit() — always.
2️⃣ It runs again every time any bound data inside projected content changes.

💡 So if the parent updates the content that was projected, this hook fires automatically.

ngAfterContentInit → Runs once, when the projected content is first rendered.
“The content just arrived; I can access it now.”

ngAfterContentChecked → Runs after every change detection that may alter projected content.
“The content might have changed; I’ll check and react if needed.”

Will it Works (Even Without ngAfterContentChecked) ?

YES, This happens because of Angular’s Change Detection mechanism — not because of lifecycle hooks directly.

When ngAfterContentChecked() actually matters ?

ngAfterContentChecked() runs after each change detection cycle, and it’s only needed if the child wants to:

Detect changes inside the projected content (not owned by the child)

React programmatically (e.g., compare old vs new values using ElementRef)

Parent Component : Passing messages to child component

         

Child Component : Displaying messages
