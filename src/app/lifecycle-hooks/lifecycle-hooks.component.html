Life Cycle Hooks

             

Why Life Cycle Hooks?

Angular components are not static HTML templates — They are dynamic classes that Angular creates, updates, and destroys automatically. So, as a developer, you often need to run logic at specific moments in that component’s “life”.

  1. When a component is first created → maybe you want to fetch data.
  2. When an @Input() value changes → maybe you want to re-render something.
  3. When Angular has rendered the view → maybe you want to interact with the DOM.
  4. When the component is about to be destroyed → maybe you want to unsubscribe or clean up.
  1. constructor() → Component instance created (class-level)
  2. ngOnChanges() → Runs when Input() changes (before init)
  3. ngOnInit() → First initialization logic
  4. ngDoCheck() → Custom change detection hook
  5. ngAfterContentInit() → After projected content loaded (via ng-content)
  6. ngAfterContentChecked() → After projected content checked
  7. ngAfterViewInit() → After component's view + child views initialized
  8. ngAfterViewChecked() → After component’s view checked again
  9. ngOnDestroy() → Cleanup logic (before removal)
Lifecycle Hook When It Runs How Many Times Side Effects / Risks Main Benefits / Use Case
constructor() When the component class is created (before Angular hooks) Once DOM and bindings not yet ready; avoid logic using Angular features Used for basic initialization like dependency injection
ngOnChanges() When any @Input() value changes (before OnInit) Multiple times (on every input change) Can trigger multiple runs if parent updates inputs frequently React to parent input changes; compare old vs new values
ngOnInit() After the first ngOnChanges(), before the view renders Once DOM elements not yet available Best for API calls, variable initialization, fetching data
ngDoCheck() During every Angular change detection cycle Many times Can be expensive if used for heavy operations Custom change detection logic; fine-grained control over updates
ngAfterContentInit() After external content (<ng-content>) projected into view Once Projected content available only now, not before Initialize logic based on projected content (content projection)
ngAfterContentChecked() After every check of projected content Many times Don’t change data here (can cause infinite loop) React when projected content updates dynamically
ngAfterViewInit() After component’s own view & child views are initialized Once Changing data here can cause ExpressionChanged error Safe place to access DOM elements & child components
ngAfterViewChecked() After every check of the component’s view Many times Never modify data here (causes infinite loops) Observe or measure DOM changes; trigger animations
ngOnDestroy() Just before the component is removed from the DOM Once If skipped, memory leaks can occur (subscriptions, timers) Cleanup: unsubscribe, clear intervals, remove listeners

Why They’re So Important :

Think of Angular as a movie director, and your component as an actor. The lifecycle hooks are like cues in the script — they tell your actor what to do and when.


| Hook                | When it runs                 | Why you use it                   |
| ------------------- | ---------------------------- | -------------------------------- |
| `ngOnInit()`        | After component creation     | Fetch API data, initialize state |
| `ngOnChanges()`     | When `@Input()` changes       | Respond to parent data changes   |
| `ngAfterViewInit()` | After view renders           | Access DOM elements, ViewChild   |
| `ngOnDestroy()`     | Before removal               | Unsubscribe, cleanup             |
| `ngDoCheck()`       | Every change detection cycle | Custom state detection logic     |
